Title (link post to a Dev.to/Hashnode technical article)

Building a unified streaming client for 26 LLM providers: why almost everyone copies OpenAI's SSE format and what that means for tooling

Body (technical write-up, not project demo)

I spent the last few months building a multi-model terminal in Rust and ran into an interesting pattern worth sharing. When you look at the API surface of 26 different LLM providers (OpenAI, Anthropic, Gemini, Mistral, DeepSeek, Groq, xAI, Together AI, Cerebras, Cohere, Fireworks, and a dozen more), 17 out of 20 cloud providers use the exact same streaming format: OpenAI-compatible SSE with `data: {json}\n\n` framing and `choices[0].delta.content` for token extraction. This means you can build ONE generic HTTP streaming client and configure it per-provider through a TOML registry instead of writing 26 separate integrations: ```rust // Same client, different base_url + API key let body = json!({ "model": model, "messages": msgs, "stream": true }); let resp = client.post(&provider.endpoint) .header("Authorization", format!("Bearer {key}")) .json(&body).send()?; // Read SSE lines, extract choices[0].delta.content ``` The three outliers: - Anthropic uses SSE but with `content_block_delta` events instead of `choices[0].delta` - Ollama uses NDJSON (newline-delimited JSON) with `message.content` - Amazon Bedrock uses AWS SigV4 signing instead of Bearer tokens The harder problem was MCP (Model Context Protocol) bridging. MCP servers communicate via JSON-RPC over stdio with LSP-style `Content-Length` headers. The challenge: making ONE tool definition work across all 26 providers. The tool descriptions get injected into the system prompt, and each provider has slightly different ideas about what a "system prompt" is (OpenAI uses `role: system`, Anthropic has a top-level `system` field, Gemini uses `systemInstruction`, DeepSeek R1 ignores system prompts entirely). The threading model was another non-obvious decision. Terminal emulation (PTY) needs a dedicated reader thread per pane. AI streaming needs to not block the PTY or the IPC thread. Solution: each PTY pane gets a `std::thread`, AI calls run in `tokio::spawn_blocking`, and the Tauri event system bridges everything to the React frontend without either side knowing about the other. The codebase is open source (Apache 2.0) if anyone wants to look at the provider registry pattern or the MCP bridge implementation: github.com/Orellius/cortex-terminal Curious if anyone else has built multi-provider LLM tooling and ran into the same convergence toward the OpenAI format.

Reddit - r/rust

Title

Built a multi-model AI terminal in Tauri 2.0 + Rust PTY - sharing the architecture and the repo

Body

Cortex Terminal is a native desktop terminal that routes queries across 26 LLM providers through a single interface. The Rust side handles everything: PTY management (portable-pty with dedicated reader threads per pane), AI provider streaming (one generic SSE client for 17+ OpenAI-compatible providers), MCP bridge (JSON-RPC over stdio), cost tracking (SQLite), and configuration (TOML). Architecture decisions worth discussing: 1. Generic provider registry: most cloud LLMs use OpenAI-compatible SSE. One streaming client handles 17+ providers. Just swap base_url + API key. Special handlers only for Anthropic (different SSE format) and local providers (NDJSON). 2. PTY threading: each terminal pane runs a dedicated reader thread. AI calls run in tokio::spawn_blocking. Neither blocks the other or the Tauri IPC thread. 3. MCP bridge: spawns MCP servers as child processes, communicates via JSON-RPC over stdio with LSP-style Content-Length framing. Tool descriptions are injected into model system prompts so all 26 providers share the same tool layer. No .unwrap() or .expect() anywhere. anyhow for app errors, thiserror for library errors. All Tauri commands return Result. Apache 2.0: github.com/Orellius/cortex-terminal Would appreciate feedback on the architecture, especially the provider registry pattern.
